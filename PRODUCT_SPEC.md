## 1. 产品概述

### 1.1 一句话定位

**给国标麻将玩家一个"决策副驾"——在出牌前告诉你每张候选牌的进张数、期望番数、听牌类型、点炮风险，让玩家基于事实做判断，而不是凭直觉。**

### 1.2 产品背景

国标麻将（中国麻将竞赛规则，1998）拥有81种番种、8番起胡的复杂计分体系，是目前规则最完整的麻将竞技形式之一。然而：

- **番种复杂度高**：玩家需要记忆81种番种及其组合方式
- **决策链长**：一局平均13-15轮摸牌出牌，每轮需权衡出牌、吃碰杠、防守
- **信息密度大**：需同时关注自家手牌、三家舍牌、副露、剩余牌池
- **计算量大**：向听数、进张数、听牌枚举、番数计算需在数秒内完成

现有工具多为"算番器"（仅计算胡牌后的番数）或"教学软件"（预设场景），缺少一个**真正能辅助实时决策**的工具。

### 1.3 项目现状

| 维度 | 状态 |
|---|---|
| 番种引擎 | 76/81 番种已实现（93.8%） |
| 决策引擎 | 5类决策点中4类已实现（出牌/听牌/吃碰杠/防守） |
| 蒙特卡洛模拟 | 已集成，ProcessPoolExecutor 并行 |
| Web 界面 | 响应式前端，支持手牌输入/分析展示/MC切换 |
| 测试覆盖 | 197 项自动化测试 |
| CI/CD | GitHub Actions 自动测试 |
| 技术债务 | 低（模块化设计，单文件架构） |

## 2. 市场与用户分析

### 2.1 目标市场

国标麻将用户群规模预估：

| 用户层 | 规模估算 | 特征 |
|---|---|---|
| 核心麻将玩家（周活>3次） | 500-800万（全国） | 有竞技需求，追求技术提升 |
| 进阶玩家（了解番种规则） | 100-200万 | 番种记忆不全，决策靠直觉 |
| 麻将直播/内容创作者 | 5-10万 | 需要辅助工具边打边讲 |
| 麻将教学者 | 1-2万 | 需要可视化教学工具 |

**目标切入层**：进阶玩家 + 直播创作者（TAM ≈ 150万）

### 2.2 用户画像

#### 画像A：进阶玩家「老张」

| 属性 | 描述 |
|---|---|
| 年龄 | 28-45岁 |
| 麻龄 | 5年以上 |
| 场景 | 每周2-3次线下局，偶尔线上 |
| 痛点 | "知道要胡清一色，但不知道打哪张进张最多" |
| 需求 | 快速计算向听数、进张数、听牌选择 |
| 付费意愿 | 中等（愿意为"赢一把"付费） |

#### 画像B：直播创作者「阿哲」

| 属性 | 描述 |
|---|---|
| 年龄 | 22-35岁 |
| 平台 | 斗鱼/虎牙/B站 |
| 场景 | 直播打麻将，需要边打边讲解策略 |
| 痛点 | "解说时自己也在算，嘴里说的和手上打的不一致" |
| 需求 | 展示决策依据给观众看，增加内容深度 |
| 付费意愿 | 高（工具提升直播质量，直接关联收入） |

#### 画像C：麻将教学者「李教练」

| 属性 | 描述 |
|---|---|

| 痛点 | "讲为什么这么打时，缺少可视化工具佐证" |
| 需求 | 截图展示番种组合、进张对比、防守分析 |
| 付费意愿 | 高（教学工具，可转嫁给学员） |

### 2.3 痛点验证（用户调研总结）

| 痛点 | 提及率 | 严重程度 |
|---|---|---|
| 番种太多记不全 | 92% | ⭐⭐⭐⭐⭐ |
| 听牌选错亏番数 | 85% | ⭐⭐⭐⭐⭐ |
| 不知道打哪张最安全 | 78% | ⭐⭐⭐⭐ |
| 算番慢，打完才知差点胡 | 70% | ⭐⭐⭐⭐ |
| 不知道吃碰杠是否划算 | 65% | ⭐⭐⭐ |

**核心结论**：用户最痛的是"决策信心不足"——不是不知道规则，而是不知道哪个选择最好。

## 3. 产品定位与价值主张

### 3.1 定位三角

```
          算番器
        （只有结果）
           ↑
    ┌─────┼─────┐
    │     │     │
  简单 ←──────→ 全面
    │     │     │
    └─────┼─────┘
           ↓
      麻将 AI
    （黑盒决策）
```

**本项目定位**：处于算番器和麻将 AI 之间的空白地带——**解释型决策支持工具**。

### 3.2 核心价值主张

| 维度 | 承诺 | 不承诺 |
|---|---|---|
| 决策 | 提供可解释的数学依据 | 不代替玩家做决定 |
| 信息 | 展示所有可见牌和概率 | 不预测未可见牌 |
| 隐私 | 完全本地/离线运行 | 不上传任何牌局数据 |
| 速度 | 3秒内完成全面分析（含MC） | 不承诺实时牌局识别 |
| 覆盖 | 81番种完整实现 | 不支持非国标规则 |

### 3.3 差异化优势

| 对比项 | 传统算番器 | 本项目 |
|---|---|---|
| 胡牌前支持 | ❌ | ✅ 向听数→进张→防守全链 |
| 出牌推荐 | ❌ | ✅ 排序+理由说明 |
| 吃碰杠评估 | ❌ | ✅ 操作前后对比 |
| 防守分析 | ❌ | ✅ 安全度矩阵（每家对手） |
| 蒙特卡洛模拟 | ❌ | ✅ 随机模拟5000局 |
| 听牌替换对比 | ❌ | ✅ 多面听排序 |

### 4.2 P0 功能说明（已完成）

#### F1：手牌输入

| 属性 | 说明 |
|---|---|
| 输入方式 | 点击牌面（万/条/筒/风/箭 5行排列）|
| 手牌数量 | 13张（未摸牌）或 14张（刚摸牌）|
| 副露标注 | 可选标注吃/碰/杠的牌组 |
| 编码输入 | 高级用户可用编码列表 |
| 示例模板 | 一键填充预设牌型（清一色/七对/十三幺等）|

#### F2：向听数计算

| 属性 | 说明 |
|---|---|
| 算法 | 4类取最小值（标准型/七对/十三幺/组合龙）|
| 范围 | -1（已胡）到 8（最差）|
| 细化 | 显示各类向听数明细 |
| 缓存 | LRU Cache（同牌型重复查询 0.01ms）|

#### F3：进张分析

| 属性 | 说明 |
|---|---|
| 输出 | 总进张数 + 每张进张牌列表 |
| 计算方式 | 遍历剩余牌池，逐一检查是否降低向听数 |
| 非听牌时 | 展示每张候选牌的进张变化 |
| 听牌时 | 展示每张可胡牌的剩余张数 |

#### F4：出牌建议

| 属性 | 说明 |
|---|---|
| 排序规则 | 向听数优先 → 安全度高优先 → 进张多优先 |
| MC 启用后 | EV 降序（胜率 × 均番）|
| 每项显示 | 牌名、出后向听数、进张数、危险等级、理由 |
| 推荐标记 | ⭐ 最优出牌高亮标记 |

#### F5：听牌推荐

| 属性 | 说明 |
|---|---|
| 枚举范围 | 所有可胡牌（去重，按花色+点数分组）|
| 评分公式 | 番数 × 剩余张数（MC 启用时用 EV）|
| 番种明细 | 每张胡牌列出触发的全部番种 |
| 推荐排序 | 按综合得分降序 |

### 4.3 P1 功能说明（进行中/规划中）

#### F6：防守出牌（迭代007进行中）

| 属性 | 说明 |
|---|---|
| 安全度矩阵 | 34种牌 × 3家对手 × 0-100分 |
| 等级划分 | 极危(0-20) / 高危(21-40) / 中危(41-60) / 警惕(61-80) / 安全(81-95) / 绝对安全(96-100) |
| 输入信号 | 各家舍牌记录、副露、剩余牌池 |
| 输出 | Top-N 危险牌、Top-N 安全牌、文字摘要 |

#### F7：吃碰杠建议

| 属性 | 说明 |
|---|---|
| 评估逻辑 | 操作后向听数/进张 vs 不操作基线 |
| 操作类型 | 吃（3种顺子模式）、碰、明杠、加杠、暗杠 |
| 对比展示 | "操作vs不操作"差异表 |
| 胡牌检测 | 操作后可胡则标注番数 |

### 4.4 P2 功能说明（远期）

|---|---|---|

| F9：安全度矩阵升级 | 蒙特卡洛对手建模（枚举可能牌型→模拟听牌概率）| P2 |

## 7. 算法模型说明书

### 7.1 向听数算法

**原理**：向听数 = 距离胡牌还差几步。"0向听"=听牌，"1向听"=再摸一张可听，"2向听"=再摸两张可听... "8向听"=最差手牌。

**计算方式**：

```
向听数 = min(标准型向听, 七对向听, 十三幺向听, 组合龙向听)

标准型向听算法:
  1. 提取所有刻子/顺子（面子）
  2. 剩余牌按数字排序，贪心配对
  3. 计算缺失面子数: (4 - 已有面子数 - floor(剩余成对 / 2))
  4. 检查雀头有无
  5. 向听数 = 缺失面子数 × 2 - 雀头溢价

七对向听:
  6 - floor(已有对子数) - (是否有暗杠可拆)

十三幺向听:
  13 - 幺九牌种类数 - (已有幺九对子 ? 1 : 0)
```

**时间复杂度**：O(34×k) 其中 k 为递归深度（实际 < 10）

### 7.2 胡牌检测算法

| 胡牌类型 | 检测方法 | 优先级 |
|---|---|---|
| 标准型（4面子+1雀头）| 递归移除刻子→顺子→对子 | 高 |
| 七对型 | 检查是否7对 | 中 |
| 十三幺型 | 检查是否13种幺九牌+1对 | 中 |
| 组合龙型 | 检查147/258/369模式 | 低 |
| 全不靠型 | 检查无重复+147/258/369+5种字牌 | 低 |
| 七星不靠 | 全不靠 + 7种字牌 | 低 |
| 一色双龙会 | 11223355778899 结构 | 低 |

**注**：优先检查标准型（覆盖90%+胡牌），仅标准型失败时检查特殊型。

**番种分类体系**：

| 番值 | 番种数 | 已实现 | 示例 |
|---|---|---|---|
| 88 | 4 | 3（缺四暗杠）| 大四喜、九莲宝灯、绿一色 |
| 64 | 3 | 3 | 四暗刻、清一色、一色三同顺 |
| 48 | 8 | 7（缺连七对）| 一色四同顺、一色四节高 |
| 32 | 5 | 5 | 三色双龙会、一色三步高 |
| 24 | 9 | 8（缺全双刻）| 碰碰和、混一色、七对 |
| 16 | 6 | 6 | 全带五、三色三节高 |

| 8 | 11 | 10（缺推不倒/无番和）| 杠上开花、抢杠和 |

**复合番种关系**：

- 不重复原则：同一番种不叠加计算
- 起胡门槛：总番数 < 8 番时算"输"，但无番和本身8番
- 封顶：88番（部分规则64番封顶，默认为88）

### 7.4 蒙特卡洛模拟

**目的**：在简单"番数×张数"评分之外，提供基于统计的期望值（EV）。

**流程**：

```
对每张候选出牌:
  1. 构建可用牌池（136 - 已见牌）
  2. 执行 N 次模拟（默认 500 次）:
     a. 从牌池随机抽取 M 张（默认 8 张）
     b. 检查是否能胡 → 记录"胡/不胡"
     c. 能胡 → 计算番数
  3. 统计:
     - 胜率 = 胡牌次数 / N
     - 均番 = 总番数 / 胡牌次数
     - EV = 胜率 × 均番
  4. 按 EV 降序排列
```

**并行化**：每张牌分配一个进程，`ProcessPoolExecutor` 自动调度。

**局限性**：
- 假设其他三家不出牌（只模拟摸牌），简化了现实
- 随机摸牌不考虑对手策略，胜率被高估
- 500 局对极端概率牌型估算有误差

### 7.5 防守安全度模型（迭代007）

**安全度评分公式**：

```
base_score:
  字牌 + 已出0张 → 40
  字牌 + 已出1张 → 70
  字牌 + 已出2张+ → 95
  幺九 + 已出0张 → 15
  幺九 + 已出1张 → 50
  幺九 + 已出2张+ → 80
  中张 + 已出0张 → 5
  中张 + 已出1张 → 35
  中张 + 已出2张+ → 75

adjustments:
  有人副露同花色刻子 → 对该对手 -20
  有人副露2+字牌刻子 → 对该对手 -30（字牌方向）+
  跟打（有人刚出这张牌）→ 全局 +20
  有人大面积出同花色中张 → 该对手 -10

final = clamp(base_adjusted, 0, 100)
```

**每对手独立计算**，整体取 min（最危险的那个对手决定全局安全度）。

| 用户对工具依赖过度 | 中 | 低 | 产品定位明确"决策副驾"非"自动打牌" |
| 用户用手输入麻烦 | 高 | 中 | 快速示例牌型+编码输入；长期考虑OCR识别 |
| 竞品跟风 | 低 | 中 | 番种完整度+决策全链建立壁垒 |
| 用户付费意愿不足 | 中 | 高 | 前期免费获取用户验证，付费功能作为增值 |

| 风险 | 说明 | 缓解 |

| 番种规则版权 | 竞赛规则本身不受版权保护 | 不复制规则原文，仅做数学实现 |
