# 国标麻将决策助手 — 产品规划 v0.2

> 类比 TXPokerAssist 经验，向国标麻将迁移。TXPokerAssist 已证明"纯数学 + 简洁 Web UI + 公开本地部署"这条路走通。麻将的复杂度更高（状态空间更小但决策点更连续、信息结构更复杂），可借鉴的工程经验很多。
>
> **当前基线（2026-06-03）**：76/81 番种实现，197 项测试，蒙特卡洛模拟引擎已集成，Web API + 前端在线。以下规划面向 Phase 2 — 从"能跑"到"好用"。

---

## 1. 项目愿景

**一句话**：给国标麻将玩家一个"决策副驾"——在出牌前告诉你每张候选牌的进张数、期望番数、听牌类型、点炮风险，让玩家基于事实做判断，而不是凭直觉。

**不是**：代替玩家决策 / 自动出牌机器人 / 算番器。

**核心价值主张**：
- **决策可解释**：每个建议附带数学依据
- **信息透明**：所有可见牌、剩余张数、期望值都展示
- **离线本地**：不上传牌局，保护隐私

## 2. 领域核心复杂度（vs 德州扑克）

| 维度 | 德州 (TXPokerAssist) | 国标麻将（本项目） | 影响 |
|---|---|---|---|
| 玩家数 | 2–10 弹性 | **固定 4** | 范围建模维度固定 |
| 牌池 | 52 张，4 花色 | **136 张**（34 种 ×4） | 枚举空间大 2.6× |
| 决策粒度 | 4 阶段（preflop/flop/turn/river） | **连续 13 轮摸牌+出牌** | 时序复杂度更高 |
| 信息结构 | 完全隐藏 + 公共牌 | **手牌隐藏 + 已出牌公开** | 部分信息博弈更复杂 |
| 胜负判定 | 1 次 showdown | **1 次胡牌**（任意玩家胡即结束） | 多人同时听牌博弈 |
| 计分方式 | pot odds（连续值） | **番数制**（8 番起胡，最高 88 番） | 离散分值，策略非线性 |
| 决策点 | 4 个静态点 | **5 类动态决策**（见 §3） | 需多引擎协同 |
| 策略深度 | 范围 vs equity | **向听数 → 听牌 → 番数 → 风险** | 多层次优化 |

**关键差异**：
1. **麻将决策是连续的**——不像德州分阶段，需要从向听数→听牌→胡牌的渐进式分析
2. **信息博弈比德州更复杂**——德州 1vN 信息博弈，麻将 4 人信息博弈 + 已出牌全公开
3. **番数制不是连续函数**——番种是离散的 1/2/4/6/8/12/16/24/32/48/64/88，导致"听 88 番 vs 听 8 番"的策略完全不同
4. **有"自摸 vs 点炮"双重风险**——自摸赢 3 家分，点炮只输 1 家，但点炮给大牌可能一炮清场
5. **摸牌规则决定了概率更可控**——136 张牌，摸过的牌不会再现，信息增量是线性的

## 3. 核心决策点（5 类）

| 决策点 | 触发时机 | 决策目标 | 数学模型 | MVP 阶段 |
|---|---|---|---|---|
| **A. 出牌选择** | 摸牌后（14→13 张） | 最小化向听数 / 最大化期望番数 | 向听数计算 + 进张枚举 + MC EV | **P0 ✅ 已实现** |
| **B. 听牌推荐** | 听牌时（0 向听） | 听哪张收益最高（番数 × 张数 × 自摸概率） | 听牌型枚举 + 番数计算 + MC | **P0 ✅ 已实现** |
| **C. 吃/碰/杠** | 他家出牌 | 是否改变手牌结构 | 改听后向听数/番数变化对比 | **P1** |
| **D. 防守出牌** | 中后期（剩余 < 50 张） | 避免点炮给大牌 | 安全度矩阵（字牌/中张/尖张 + 各家舍牌追踪） | **P1** |
| **E. 查叫预估** | 听牌后 | 估计其他三家的听牌可能性 | 蒙特卡洛对手建模 | **P2** |

| 决策引擎 | 状态 | 备注 |
|---|---|---|
| A. 出牌选择 | ✅ 已实现 | `game_engine.py` + `shanten.py`，支持 MC 排序 |

| E. 查叫预估 | ❌ 未实现 | 依赖 MC 框架扩展 |

## 8. 算法架构

### 8.1 模块依赖图

```

     ↓

     ↓
shanten.py                        # 向听数计算（4 类取最小 + 缓存）
     ↓
fan_calculator.py                 # 番数计算（81 番种检查 + 缓存）
     ↓
listen_engine.py                  # 听牌推荐引擎（枚举可胡牌 + 番数计算 + 排序）
     ↓
monte_carlo.py                    # 蒙特卡洛模拟（随机摸牌 N 轮 → 统计 EV）
     ↓
game_engine.py                    # 综合决策引擎（整合全部模块）
     ↓
api/main.py                       # Web API（FastAPI）
```

### 8.2 向听数算法

```
向听数 = min(标准型向听, 七对向听, 十三幺向听, 组合龙向听)

标准型向听算法（核心）:
  1. 尝试移除所有刻子/顺子（杠→刻处理）
  2. 剩余牌按数字排序，贪心配对
  3. 差数 = 缺失的面子数（3 张一组 - 已有组数）
  4. 补正：是否有雀头 / 是否需要雀头
  5. 向听数 = 最大(diff, 0)

特殊型向听:
  七对: 对子数向 7 补足
  十三幺: 幺九牌种数 + 对子有无
  组合龙: 147/258/369 模式匹配后的缺失数
```

  #    a. 从牌池随机分配后续摸牌

## 11. 技术栈

| 层 | 选型 | 理由 |
|---|---|---|
| 后端框架 | FastAPI（`api/main.py`） | 原生异步 + 自动文档 + 类型提示 |
| 算法 | 纯 Python 标准库 | 零外部依赖，数学运算无需 numpy |
| MC 引擎 | Python `concurrent.futures.ProcessPoolExecutor` | 充分利用多核 |
| 缓存 | `functools.lru_cache` | 手牌 hash 命中率高 |
| 前端 | HTML + CSS + JS（无框架） | 零依赖，轻量 |
| 部署 | Vercel + FastAPI WSGI 适配 | 已验证可行（`runtime.txt` + `vercel.json`） |

## 13. 关键工程决策（ADR）

### ADR-001：番数计算用 LRU 缓存而非预计算表

- **决策**：`functools.lru_cache(maxsize=5000)` 动态缓存
- **理由**：手牌组合实在太多（C(34,14) ≈ 2×10^9），无法预计算
- **效果**：第二次起同牌型分析命中率 90%+

### ADR-002：蒙特卡洛模拟用 ProcessPoolExecutor 而非 async/await

- **决策**：CPython GIL 限制，CPU 密集型任务用多进程
- **理由**：每张牌模拟 1000 局，独立无状态，完美并行
- **效果**：8 核机器 ≈ 5 倍加速

### ADR-003：Web 框架选 FastAPI 而不是纯 WSGI

- **决策**：FastAPI → Uvicorn（本地开发）→ Vercel WSGI 适配
- **理由**：
  - 本地开发热加载体验好
  - Pydantic 模型校验自动处理
  - Vercel 部署只需在 `api/index.py` 中 wsgi 封装
- **成本**：`fastapi` 包在 Vercel 上约 200KB

### ADR-004：防守安全度分 0-100 而非概率值

- **决策**：安全度 = 基于经验规则的打分，不是精确概率
- **理由**：对手手牌信息不完整，精确概率不可计算
- **输入**：各家舍牌记录、字牌出现数、中张剩余数
- **局限性**：不建模对手听牌偏好，后续可用 MC 扩展

## 16. 已知风险与缓解

| 风险 | 概率 | 影响 | 缓解 |
|---|---|---|---|
| 防守安全度不准导致误判 | 中 | 高 | 默认标注"推理中不确定性"，不替用户决定 |

| 番种遗漏/番值偏差 | 中 | 中 | 逐番对照官方规则手册，每个番种独立测试 |
| 前端牌面交互不直观 | 中 | 中 | 用户测试后快速迭代 UI，MVP 以功能性优先 |
| 8 番起胡规则误判 | 低 | 高 | `is_valid` 字段必须独立于番数计算器 |
